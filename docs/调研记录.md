
# Sisyphus-Y 平台调研结论与改版需求（2026-03）

## 一页摘要

本次调研的结论不是推翻平台，而是把现有能力收敛到一条更清晰的主链路：**分析台 → 工作台 → 用例库**。平台已经具备 Prompt 管理、`glm-5`、UDA 结构化解析、XMind 导入、三层目录树、RAG 阈值 0.72 等点状能力，但核心使用体验仍存在主链路割裂、页面职责重叠、文档与实现不同步的问题。

本轮改版必须优先完成以下 5 件事：

1. **工作台双步骤化**：统一为 `Step 1 确认测试点` 与 `Step 2 生成用例` 的同页流程，允许从 Step 2 回退到 Step 1 继续补点。
2. **分析域统一入口**：以 `/analysis` 为主入口，兼容旧页面，但新增实现优先收敛到分析域，不再继续扩散独立页面职责。
3. **AI 配置一致化**：将“模型列表 / Prompt 管理 / 生效配置”对齐为统一体验，避免“后端已有能力、前端仍是旧表单”的割裂。
4. **历史用例质检闭环**：清洗后的历史用例必须先经过审查，再入向量库；必要时清理旧向量，避免脏数据重复召回。
5. **计划与验收分离**：`process.json` 负责本轮开发迭代计划，`progress.json` 继续保留为验收/回归记录，不混用。

## 当前代码状态对照

| 维度 | 已落地能力 | 本轮待补齐 |
| --- | --- | --- |
| 导航与页面框架 | 9 个一级菜单、顶部统一导航、仪表盘双 Tab、设置页 Prompt 管理 | 分析域路由收敛、旧页面去重、帮助入口与首次引导整合 |
| 分析链路 | 分析页已具备需求列表、AI 分析、覆盖追踪三 Tab 结构 | `/analysis` 与旧 `/diagnosis` / `/scene-map` / `/coverage` 仍并存，需统一心智 |
| 工作台链路 | 已有需求导航、对话式生成、SSE 持久化、已生成用例侧栏 | 缺少 Step 1 测试点确认，生成默认仍偏“旧会话流”而非“已确认测试点流” |
| AI 配置与 Prompt | Prompt 历史、默认值/回滚、`glm-5`、provider 列表、`/uda/parse-structure` 已具备 | 模型列表管理与前端 UI 尚未完全对齐，仍有“旧全局配置表单”痕迹 |
| 历史用例/RAG | RAG top-k=5、阈值 0.72、XMind 导入、目录树已具备 | “清洗 → 审查 → 润色 → 重新入向量库 → 报告” 闭环仍需串起来 |

## 推荐实施顺序

1. 先对齐需求文档、README、CHANGELOG、全局加载文档，统一目标与术语。
2. 写入 `process.json`，明确每个迭代周期的目标、依赖、验收口径与状态。
3. 第一批代码优先攻主链路：分析台与工作台之间的状态衔接、工作台双步骤化、仅使用“已确认测试点”参与生成。
4. 第二批补齐 AI 配置一致化、RAG/历史用例质检闭环、仪表盘/引导完善。
5. 最后用 `progress.json` 回写验收项、证据与回归结果。

**C1 逐条回答：**

**1. 历史用例质量清理**

不要直接入库，必须先经过质量审查。执行方式如下：

启动多个并行 subagent，调用 `/functional-test-case-reviewer` skill，对 `[清洗后数据]` 目录下的所有用例逐一审查。每个 subagent 负责一个子目录，并行处理提升效率。审查标准严格执行以下规则，不达标的用例不允许直接入库：

- 第一步必须是「进入 [完整页面路径] 页面」，路径精确到具体功能页
- 每个步骤有对应预期结果，多个预期用 `1) 2) 3)` 格式
- 前置条件涉及数据库的必须有完整建表语句 + 插入数据
- 步骤完全独立，不依赖其他用例的执行结果
- 预期结果具体可验证，不出现「操作成功」「显示正确」「无报错」

处置规则：

- 审查通过 → 直接入向量库
- 有问题但可修复 → subagent 自动润色修改后再入库
- 无法修复（信息严重缺失）→ 标记丢弃，记录原因，不入库

审查完成后输出报告：总条数 / 通过数 / 润色修改数 / 丢弃数 / 丢弃原因汇总。

**2. RAG 检索 top-K**

取 **5 条**。其中相似度阈值设为 0.72，低于阈值的不强行凑数，宁可少于 5 条也不引入低相关噪声。检索结果注入 Prompt 时带上相似度分数，让模型自己判断参考权重。

**3. Prompt 重写**

不需要全部推翻，在现有 7 层结构基础上补充两类示例：

- **Few-shot 正例**：从审查通过的高质量历史用例中各取 1 条（覆盖正常/异常/边界三种场景类型），作为格式和质量基准
- **负面示例**：各取 1 条典型的低质量用例（模糊预期、步骤不独立、前置条件缺失），明确标注「禁止生成此类内容」

Few-shot 示例直接从 `[清洗后数据]` 审查通过的用例里挑，不要手写。

**4. GLM-5 模型 ID**

模型 ID 使用 `glm-5`，这是目前智谱开放 API 中能力最强、稳定性最高的版本。Base URL 不变。

你说得对，两个页面的职责高度重叠。区别在于：仪表盘是项目级概览（我的项目整体怎么样），质量看板原本想做迭代级深度分析（这次迭代用例质量分布、AI 生成效率统计）。但这个差异不足以支撑单独一个菜单项。

合并方案：仪表盘页面增加一个「质量分析」Tab，内容包含原质量看板的所有图表：

```
仪表盘页面
├── Tab：概览（原仪表盘内容，统计卡片 + 趋势折线图）
└── Tab：质量分析（迭代用例质量分布 + AI 生成效率 + 覆盖率健康度）
```

左侧菜单删除「质量看板」入口，只保留「仪表盘」，点进去默认展示「概览」Tab。菜单项从原来的 10 个减少到 9 个，导航更清爽。
